# xray-checker-statuspage

Современная кастомная страница статуса VPN/прокси-серверов поверх
[xray-checker](https://github.com/kutovoys/xray-checker).

Строка на каждый сервер, шкала аптайма за 30 дней с флагами стран; наведение на день —
дата, аптайм и длительность простоя. Клик по любому дню раскрывает детализацию: график пинга
на логарифмической шкале (норма и редкие всплески видны одновременно, значение линии всегда
совпадает с подсказкой) и периоды сбоев красными полосами.
Активное окно 12 часов с прокруткой, «сейчас» по центру. Тёмная/светлая тема, шрифт Inter с
кириллицей (локально), своё лого и favicon.

Чат и поддержка: https://t.me/+O5jAhwcYdYhlY2Yy

xray-checker  →  реально тестит прокси через Xray (порт 2112, внутренний)

## Возможности

- Аптайм за 30 дней по каждому серверу, флаги стран, общий простой.
- Детализация дня: график пинга, периоды сбоев, пики, статистика проверок.
- «Подтверждённый» расчёт простоя (одиночный блип не считается за целый интервал).
- Тёмная/светлая тема по системной, плавные анимации, адаптив под мобильные.
- Своё название, лого и favicon; история в SQLite (лёгкая, авто-очистка).
- Анти-фингерпринт: рандомизация классов и маскировка Server-заголовка.

## Установка

Нужны Ubuntu/Debian и root. Установщик сам поставит Docker при необходимости.

Рабочую папку `/opt/xray-checker-statuspage` создаёт сам установщик (через `git clone`) —
вручную создавать её не нужно. Если папка уже существует, установщик подключит её к репозиторию,
не трогая твои `docker-compose.yml` и `data/`. Папку `data/` (база, шрифты) создаёт
установщик/Docker автоматически.

### Вариант A. Чистый сервер (установщик сам поставит nginx + HTTPS)

Если на сервере ничего нет и порты 80/443 свободны:

```bash
curl -fsSL https://raw.githubusercontent.com/Mrvibecodic/xray-checker-statuspage/main/install.sh -o install.sh
sudo bash install.sh
```

На вопрос про nginx отвечаешь «да» и вводишь домен. nginx ставится из официального
репозитория **nginx.org** (только если его ещё нет на сервере). Затем выбираешь способ
сертификата:

- **HTTP-01** — проверка по 80 порту (порт должен быть открыт извне);
- **Cloudflare DNS** — без открытия порта; подойдёт API Token (права `Zone.DNS: Edit`) или
  Global API Key + email аккаунта Cloudflare;
- **без сертификата** — nginx по HTTP.

Если сертификат не выпустится (закрыт 80 порт, неверный доступ к Cloudflare) — установка не
прервётся: страница останется на HTTP, выпуск можно повторить позже.

### Вариант B. Сервер с панелью или уже установленным nginx

Свой веб-сервер ставить не нужно — проксировать будет панель или твой nginx. На вопрос про
nginx отвечаешь «нет». Сервис поднимется на `http://127.0.0.1:8080`.

```bash
curl -fsSL https://raw.githubusercontent.com/Mrvibecodic/xray-checker-statuspage/main/install.sh -o install.sh
sudo bash install.sh
```

Дальше направляешь трафик на него:

- **Панель (FastPanel/ISPmanager):** создай сайт на домене → обратный прокси (reverse proxy) на
  бэкенд `http://127.0.0.1:8080`. Сертификат — средствами панели. Убедись, что HTTP-авторизация
  на сайте выключена.
- **Готовый nginx (вручную):** добавь server-блок (замени `status.your-domain.example` на свой
  домен):

  ```nginx
  server {
      listen 80;
      server_name status.your-domain.example;

      location / {
          proxy_pass http://127.0.0.1:8080;
          proxy_set_header Host $host;
          proxy_set_header X-Real-IP $remote_addr;
          proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
          proxy_set_header X-Forwarded-Proto $scheme;
      }
  }
  ```

  Сохрани в `/etc/nginx/sites-available/status.conf`, включи и перезагрузи:

  ```bash
  ln -s /etc/nginx/sites-available/status.conf /etc/nginx/sites-enabled/
  nginx -t && systemctl reload nginx
  certbot --nginx -d status.your-domain.example
  ```

### Вариант C. Вручную, без скрипта

```bash
git clone https://github.com/Mrvibecodic/xray-checker-statuspage /opt/xray-checker-statuspage
cd /opt/xray-checker-statuspage
cp docker-compose.example.yml docker-compose.yml
nano docker-compose.yml      # вписать SUBSCRIPTION_URL, METRICS_PASSWORD и т.д.
docker compose up -d --build
```

## Обновление

```bash
sudo bash /opt/xray-checker-statuspage/install.sh
```

Если конфиг уже есть — выбери «не перенастраивать»: скрипт подтянет свежий код с гита и
пересоберёт контейнеры, настройки сохранятся.

## Обновление подписки (добавлен новый сервер)

Ничего делать вручную не нужно. xray-checker сам перечитывает подписку каждые
`SUBSCRIPTION_UPDATE_INTERVAL` секунд (по умолчанию 300), и страница подхватывает новые
серверы на следующем опросе — обычно в течение нескольких минут. Новый сервер появится строкой
в списке; его 30-дневная история начнёт копиться с момента появления (более ранние дни будут
серыми — «нет данных»). Убранный из подписки сервер так же сам пропадёт со страницы.

Если ты **сменил саму ссылку подписки** (новый URL), поправь `SUBSCRIPTION_URL` в
`docker-compose.yml` и примени:

```bash
cd /opt/xray-checker-statuspage
docker compose up -d
```

## Кастомизация

Всё задаётся переменными окружения в `docker-compose.yml`. После правки — `docker compose up -d`.

Сервис `statuspage`:

| Переменная | По умолчанию | Описание |
|---|---|---|
| `TITLE` | `Статус серверов` | заголовок в шапке и во вкладке браузера |
| `SUBTITLE` | — | подзаголовок под названием |
| `TZ` | `Europe/Moscow` | часовой пояс дат |
| `DAYS` | `30` | сколько дней истории показывать |
| `POLL_INTERVAL` | `300` | период опроса checker, сек (держи = `PROXY_CHECK_INTERVAL`) |
| `SAMPLE_RETAIN_DAYS` | `DAYS+1` | сколько дней хранить поминутные данные (детализация по дням) |
| `SERVER_HEADER` | `nginx` | значение HTTP-заголовка `Server` (маскировка) |
| `CHECKER_URL` | `http://xray-checker:2112` | адрес xray-checker |
| `DB_PATH` | `/data/status.db` | путь к базе |

Сервис `xray-checker` (основные):

| Переменная | По умолчанию | Описание |
|---|---|---|
| `SUBSCRIPTION_URL` | — | ссылка на подписку (можно несколько через запятую) |
| `PROXY_CHECK_INTERVAL` | `300` | как часто реально тестируются прокси, сек |
| `PROXY_TIMEOUT` | `30` | таймаут одной проверки, сек |
| `PROXY_CHECK_METHOD` | `ip` | метод проверки: `ip`, `status` (легче), `download` |
| `METRICS_USERNAME` / `METRICS_PASSWORD` | — | Basic Auth для админки/метрик checker |

### Логотип и favicon

Положи картинку с именем, начинающимся на `favicon`, в папку `data` рядом с
`docker-compose.yml` — сервис сам подхватит её и как favicon (вкладка браузера), и как лого в
шапке. Пересборка не нужна, просто обнови страницу.

- Поддерживаемые форматы: `png`, `svg`, `ico`, `jpg`, `webp` (например `data/favicon.png`).
- Рекомендуется квадратная картинка, например PNG 256×256 (лого в шапке обрезается под квадрат,
  так что квадрат смотрится лучше всего). SVG — идеально для чёткости на любом экране.
- Нет файла — остаётся дефолтный значок-щит.

### Тема и оформление

Светлая/тёмная тема переключается автоматически по настройке браузера/системы — отдельно
включать не нужно. Цвета простоя: зелёный (без сбоев), жёлтый (до 30 мин), оранжевый
(30 мин – 2 ч), красный (от 2 ч).

## Частота опроса

Чтобы проверять чаще, уменьши оба интервала и держи их равными:

```yaml
  xray-checker:
    environment:
      - PROXY_CHECK_INTERVAL=60
  statuspage:
    environment:
      - POLL_INTERVAL=60
```

- Проверки идут параллельно, поэтому цикл ≈ один таймаут (`PROXY_TIMEOUT`, по умолчанию 30 сек).
- Жёсткое правило: `PROXY_CHECK_INTERVAL` > `PROXY_TIMEOUT`. При таймауте 30 сек минимум ~40–60 сек.
- Для интервала 15–20 сек снизь и `PROXY_TIMEOUT` (например до 10 сек).
- Практический минимум для большинства — 30–60 сек.

## Как считается простой

Чтобы одиночный неудачный опрос не записывался как целый интервал, простой считается только по
подряд идущим неудачным проверкам: 1 сбой → 0 (кратковременный блип), 2 подряд → 1 интервал,
3 → 2 интервала и т.д. Процент аптайма при этом учитывает все проверки, поэтому одиночный блип
виден в проценте и в подсказке как «кратковременный сбой».

## Нагрузка на SQLite и ёмкость

База хранит дневные агрегаты (крошечные — на них 30-дневная шкала) и поминутные замеры за
`SAMPLE_RETAIN_DAYS` дней (детализация по дням). Размер определяют замеры (~70 байт каждый).
При хранении 31 день:

| Серверов | Интервал 60с | 30с | 15с | 10с |
|---|---|---|---|---|
| 10 | ~31 МБ | ~62 МБ | ~125 МБ | ~187 МБ |
| 25 | ~78 МБ | ~156 МБ | ~312 МБ | ~470 МБ |
| 50 | ~156 МБ | ~312 МБ | ~625 МБ | ~940 МБ |
| 100 | ~312 МБ | ~625 МБ | ~1.25 ГБ | ~1.9 ГБ |

Производительность SQLite тут не ограничитель (выборка за день по индексу быстра даже при
десятках млн строк) — упирается в место на диске. Хочешь сэкономить — уменьши
`SAMPLE_RETAIN_DAYS`; 30-дневная шкала аптайма не пострадает.

## Приватность

Страница отдаётся с `noindex, nofollow` (не попадает в поисковики), заголовок `Server`
маскируется (`SERVER_HEADER`), а имена CSS-классов рандомизируются при каждом старте контейнера —
сложнее найти инстансы по шаблонным «тегам».
