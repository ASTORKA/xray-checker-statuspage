# Статус-пейдж VPN/прокси-серверов (probe-only)

Современная страница статуса VPN/прокси-серверов: показывает, **реально ли работает
каждый сервер у пользователей в зоне блокировки (РФ)**, а не просто доступен ли он из
облака. Живой статус считается по данным **агентов-пробников**, поставленных на
устройства в РФ, — они через `xray-core` поднимают реальный туннель к каждому серверу
из подписки и проверяют, проходит ли трафик сквозь DPI.

Строка на каждый сервер, шкала аптайма за 30 дней с флагами стран; наведение на день —
дата, аптайм и длительность простоя. Клик по дню раскрывает детализацию: график пинга
на логарифмической шкале и периоды сбоев красными полосами (а периоды, когда на
устройстве был включён VPN — серыми). Тёмная/светлая тема, шрифт Inter с кириллицей
(локально), своё лого и favicon.

> Исходно проект делался поверх [xray-checker](https://github.com/kutovoys/xray-checker)
> (отсюда название репозитория) — спасибо [kutovoys](https://github.com/kutovoys). В
> текущей версии xray-checker как зависимость **убран**: statuspage сам тянет и парсит
> подписку, поэтому одна подписка обновляется автоматически без рестартов.

Чат и поддержка: https://t.me/+O5jAhwcYdYhlY2Yy

![Превью интерфейса](assets/preview.png)

```
PROBE_SUBSCRIPTION_URL → statuspage сам тянет/парсит подписку (порт 8080)
                              ├── список серверов для UI (имена, флаги)
                              └── таргеты для пробников (/api/probe/targets)
probe-агент (на устройстве в РФ) → xray-core поднимает туннель к каждому серверу,
                                    проверяет проход трафика, репортит на statuspage
```

## Возможности

- Аптайм за 30 дней по каждому серверу × каждому пробнику, флаги стран, общий простой.
- Детализация дня: график пинга (лог. шкала), периоды сбоев, VPN-периоды серым.
- **Честная проверка сквозь DPI**: агент через `xray-core` поднимает туннель с тем же
  fingerprint/REALITY, что в подписке, и бьёт пачкой параллельных коннектов — ровно так
  DPI рубит заблокированный fingerprint (одиночный коннект его не ловит).
- Индикатор устройства: зелёный (онлайн) / красный (офлайн) / серый (на устройстве VPN) /
  синий (устройство неактивно).
- Авто-обновление подписки без рестартов (одна подписка — один источник).
- Тёмная/светлая тема, адаптив; история в SQLite; очистка истории из админ-режима.
- Анти-фингерпринт: рандомизация классов и маскировка Server-заголовка.

## Установка

Нужны Ubuntu/Debian и root. Установщик сам поставит Docker при необходимости.

Установщик клонирует исходники этого форка в `/opt/xray-checker-statuspage/src` и собирает
Docker-образ локально (`docker compose up -d --build`). Рабочую папку, `data/` и `src/`
создаёт сам установщик; вручную создавать ничего не нужно. Образ собирается быстро —
`python:3.12-slim` + один `app.py`.

> **Важно: отключите HWID у сервисного пользователя.** Учётка, чью ссылку подписки вы
> укажете в `PROBE_SUBSCRIPTION_URL`, должна быть **без ограничения HWID** (лимита
> устройств). Иначе при включённом HWID панель может отдать заглушку вместо списка
> серверов. В карточке этого пользователя в панели выключите HWID (или заведите
> отдельного служебного юзера без лимита устройств).

### Вариант A. Чистый сервер (установщик сам поставит nginx + HTTPS)

```bash
curl -fsSL https://raw.githubusercontent.com/ASTORKA/xray-checker-statuspage/main/install.sh -o install.sh
sudo bash install.sh
```

На вопрос про nginx отвечаешь «да» и вводишь домен. nginx ставится из официального
репозитория **nginx.org** (только если его ещё нет). Сертификат HTTPS обязателен, на выбор:

- **HTTP-01** — проверка по 80 порту (порт открыт извне);
- **Cloudflare DNS** — без открытия порта; API Token (`Zone.DNS: Edit`) или Global API Key + email.

### Вариант B. Сервер с панелью или уже установленным nginx

На вопрос про nginx отвечаешь «нет» — сервис поднимется на `http://127.0.0.1:8080`, проксирует панель/твой nginx.

```bash
curl -fsSL https://raw.githubusercontent.com/ASTORKA/xray-checker-statuspage/main/install.sh -o install.sh
sudo bash install.sh
```

- **Панель (FastPanel/ISPmanager):** сайт на домене → обратный прокси на `http://127.0.0.1:8080`, сертификат средствами панели. HTTP-авторизацию на сайте выключи.
- **Готовый nginx:** server-блок (замени домен):

  ```nginx
  server {
      listen 80;
      server_name status.your-domain.example;
      location / {
          proxy_pass http://127.0.0.1:8080;
          proxy_set_header Host $host;
          proxy_set_header X-Real-IP $remote_addr;
          proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
          proxy_set_header X-Forwarded-Proto $scheme;
      }
  }
  ```

  ```bash
  ln -s /etc/nginx/sites-available/status.conf /etc/nginx/sites-enabled/
  nginx -t && systemctl reload nginx
  certbot --nginx -d status.your-domain.example
  ```

## Обновление

```bash
cd /opt/xray-checker-statuspage && git -C src pull && docker compose up -d --build
```

Либо повторно `sudo bash install.sh` → «не перенастраивать»: обновит образ и перезапустит, настройки сохранятся.

## Обновление подписки (новый сервер)

Вручную ничего не нужно. statuspage сам тянет подписку и кеширует разобранные таргеты на
`PROBE_TARGETS_TTL_MIN` минут (по умолчанию 10) — новые серверы появляются на странице, а
убранные из подписки сами пропадают в течение этого окна. История нового сервера копится с
момента появления.

Если сменил саму ссылку подписки — поправь `PROBE_SUBSCRIPTION_URL` в `docker-compose.yml`
и `docker compose up -d`.

## Кастомизация

Переменные окружения в `docker-compose.yml`. После правки — `docker compose up -d`.

Сервис `statuspage`:

| Переменная | По умолчанию | Описание |
|---|---|---|
| `TITLE` | `Статус серверов` | заголовок в шапке и во вкладке |
| `SUBTITLE` | — | подзаголовок |
| `TZ` | `Europe/Moscow` | часовой пояс дат |
| `DAYS` | `30` | сколько дней истории показывать |
| `POLL_INTERVAL` | `60` | период sync подписки в список серверов UI, сек |
| `SERVER_HEADER` | `nginx` | значение заголовка `Server` (маскировка) |
| `DB_PATH` | `/data/status.db` | путь к базе |
| `ADMIN_TOKEN` | — | секрет для админ-режима (удаление записей, очистка истории, регистрация пробников). Пусто — функция выключена. |
| `STALE_AFTER_HOURS` | `24` | через сколько часов авто-удалять серверы, исчезнувшие из подписки. `0` — отключить. |
| `PROBE_SUBSCRIPTION_URL` | — | **главное:** URL подписки. Сервер сам тянет/парсит, отдаёт агентам разобранные таргеты и строит список серверов для UI. Пусто — пробники получат 503. |
| `PROBE_SUBSCRIPTION_USER_AGENT` | `v2rayN/6.40` | UA при запросе подписки (панели смотрят на UA). |
| `PROBE_TARGETS_TTL_MIN` | `10` | TTL in-memory кеша таргетов (минуты). |
| `PROBE_FRESH_MINUTES` | `10` | окно «свежести» проб — старее этого устройство считается неактивным. |
| `PROBE_SAMPLE_RETAIN_HOURS` | `(DAYS+1)*24` | сколько часов хранить поминутную историю проб. |

**Переменные агента** (env на устройстве пробника, не на сервере) — см. секцию про
пробников: `INTERVAL`, `EXPECT_COUNTRY`, `PROBE_BURST_COUNT`, `PROBE_BURST_OK_RATIO`,
`XRAY_BIN`, `PROBE_BULK_URL` и т.д.

### Логотип и favicon

Положи картинку с именем на `favicon` (`png/svg/ico/jpg/webp`) в папку `data` рядом с
`docker-compose.yml` — подхватится как favicon и лого в шапке. Рекомендуется квадрат (PNG 256×256)
или SVG. Нет файла — дефолтный значок-щит.

### Тема

Кнопка в шапке циклирует между четырьмя темами:

- **светлая** — нейтрально-холодная палитра по умолчанию;
- **тёмная** — традиционный тёмный режим;
- **Claude** — тёплая кремовая палитра (`#F0EEE6` фон, `#FAF9F5` карточки) с
  фирменным copper-оранжевым акцентом `#D97757` — в духе brand-кода Anthropic;
- **Claude Code** — тёплая тёмная палитра (`#1A1815` фон, `#262320` карточки)
  с тем же copper-оранжевым — CLI-вайб Claude Code.

Иконка кнопки показывает следующую тему в цикле (луна → искра → `>_` → солнце →…).
Выбор сохраняется в `localStorage`. При первом заходе используется тёмная/светлая по
системной настройке. Цвета простоя на барах не привязаны к теме: зелёный (нет сбоев),
жёлтый (до 30 мин), оранжевый (30 мин – 2 ч), красный (от 2 ч).

### Группировка sub-серверов по имени хоста

`statuspage` идентифицирует каждую строку подписки своим `sid`
(`"s" + sha256(host|port|sni|uuid)`). Один логический хост может быть представлен
несколькими строками подписки с одинаковым `#name` (например, роутинг между апстримами).
Чтобы не плодить дубли, `statuspage` группирует серверы по сырому `name`:

- Все sid'ы с одинаковым именем показываются **одной** строкой.
- График пинга внутри строки сливает samples всех членов группы.
- Админ-удаление (×) сносит **всю группу** целиком.

API-поле `members` в `/api/summary` показывает, сколько sub-серверов скрыто за строкой
(`members: 1` — одиночный хост; `members: 2+` — группа).

### Авточистка устаревших записей

`statuspage` идентифицирует сервер по `sid` (хеш `host|port|sni|uuid`). Когда меняешь
конфиг ноды (например, fingerprint или ключ REALITY) или хост исчезает из подписки, `sid`
меняется/пропадает, а старая запись остаётся висеть в БД офлайн навсегда. Поэтому по
умолчанию `statuspage` при каждом sync проверяет, какие `sid` всё ещё есть в свежей
подписке, и автоматически удаляет из `current`/`daily`/`samples` всё, что не приходило
больше `STALE_AFTER_HOURS` часов (по умолчанию 24). **Нюанс группировки:** группа
удаляется только если **все** её sid'ы (с тем же `name`) исчезли и старше порога.

- Чистка происходит только если подписка пришла непустой (защита от временного лага сети).
- Агрессивнее — `STALE_AFTER_HOURS=1` или меньше.
- Ручной контроль — `STALE_AFTER_HOURS=0` отключает авточистку полностью; чистить дубли
  можно вручную через × в админ-режиме.

В админ-режиме над списком серверов появляется переключатель «**Авто-удаление устаревших
записей**» — без редактирования env-переменных можно выключить/включить чистку на лету.
Состояние сохраняется в SQLite (таблица `settings`) и переживает рестарт контейнера. Если
`STALE_AFTER_HOURS=0` — тогл показывается, но недоступен (мастер-выключатель в env имеет
приоритет).

### Удаление записей со страницы (чистка дублей)

`statuspage` идентифицирует сервер по `sid` (хеш `host|port|sni|uuid`). Когда меняешь
конфиг ноды или хоста, у того же физического сервера появляется **новый** `sid`, а старая
запись остаётся висеть в БД. Со временем накапливаются дубли. Чтобы удалить дубль вручную:

1. Нужен **`ADMIN_TOKEN`** для входа в админ-режим:

   - **Через install.sh** — токен генерируется автоматически при первой установке.
     Подсмотри его в `/opt/xray-checker-statuspage/.install-info` (строка
     «Админ-токен …»). На повторной установке (или при `sudo bash install.sh` поверх
     существующего) скрипт спросит, добавить ли токен, если его ещё нет.
   - **Вручную** — добавь в `docker-compose.yml` в секцию `statuspage`:

     ```yaml
     statuspage:
       environment:
         - ADMIN_TOKEN=придумай-длинный-секрет
     ```

     и `docker compose up -d`.

2. Открой страницу — в шапке появится **иконка-замок** рядом со сменой темы. Нажми на неё,
   введи токен. Токен запомнится в `localStorage` браузера; повторный ввод не нужен.

3. В режиме админа справа от каждой строки появится кнопка **×** — клик → подтверждение →
   запись удаляется (вся история по этому серверу стирается). Там же кнопка **«Очистить
   историю опросов»** — сброс всей накопленной статистики разом (см. «Очистка истории»).

Чтобы выйти из админ-режима — снова клик по иконке-замку.

> **Если сервер вернётся после удаления** — значит, он всё ещё есть в подписке, и
> следующий sync его снова добавит. Удалять имеет смысл *старые `sid`*, которых уже нет в
> подписке; активные сервера будут возвращаться, и это нормально.

Если `ADMIN_TOKEN` не задан или пуст — иконка-замок не показывается и удаление недоступно
никому (включая владельца). Это безопасное состояние по умолчанию.

## Региональные пробники (агенты на устройствах)

Облако не видит ban'ы по фингерпринту/SNI/IP, которые активны только в РФ. Поэтому живой
статус считается **только по данным пробников** — агентов на устройствах в зоне
блокировки. Каждый агент через `xray-core` поднимает реальный туннель к серверам из
подписки и проверяет проход трафика сквозь DPI (см. «Что проверяет агент» ниже).

На странице у каждого сервера — **горизонтальные полосы, по одной на каждого
зарегистрированного пробника**. У каждой полосы своя 30-дневная шкала аптайма, текущий
пинг и точка-индикатор устройства:

| Цвет | Состояние |
|---|---|
| 🟢 зелёный | активен, сервер работает |
| 🔴 красный | активен, сервер не работает (DPI/блок) |
| ⚪ серый | на устройстве включён VPN — пробы не идут |
| 🔵 синий | устройство неактивно (давно не слало) |

**Подготовка на стороне сервера.** URL подписки задаётся при `sudo bash install.sh`
(вопрос «URL подписки»). Если ставишь вручную — добавь в `docker-compose.yml` в секцию
`statuspage`:

```yaml
statuspage:
  environment:
    - PROBE_SUBSCRIPTION_URL=https://your-panel/sub/...
```

И `docker compose up -d`. Сервер сам тянет подписку, кеширует разобранные таргеты на
`PROBE_TARGETS_TTL_MIN` минут и отдаёт их пробникам — секретный URL на устройства не
попадает.

**Поставить пробника на macOS** (нужно `python3` — установится с Xcode CLT, плюс
`ADMIN_TOKEN` со страницы):

```bash
mkdir -p ~/xrs-probe && cd ~/xrs-probe
curl -fsSL https://raw.githubusercontent.com/ASTORKA/xray-checker-statuspage/main/probes/install-macos.sh -o install-macos.sh
bash install-macos.sh
# спросит: URL статус-страницы, ADMIN_TOKEN, имя пробника
# секретный URL подписки на устройство НЕ попадает — его держит сервер
```

Что происходит:
- Установщик регистрирует пробника на сервере, получает уникальный `probe_token`.
- Кладёт `agent.py` в `~/.xrs-probe/`, конфиг — в LaunchAgent plist
  `~/Library/LaunchAgents/com.xrs.probe.plist`.
- LaunchAgent поднимает агента при логине пользователя и автоматически перезапускает,
  если он упал.
- Логи: `~/Library/Logs/xrs-probe.log` — `tail -f` чтобы посмотреть, что происходит.
- Устанавливает CLI **`monitorvpn`** (symlink в `/usr/local/bin/monitorvpn`).

**Управление через `monitorvpn`:**

```
monitorvpn start       запустить агента
monitorvpn stop        остановить
monitorvpn restart     перезапустить
monitorvpn status      статус + последние строки лога
monitorvpn logs        tail -f лога
monitorvpn refresh     попросить сервер перечитать подписку (когда ты её обновил)
monitorvpn update      обновить agent.py и сам monitorvpn (без переустановки)
monitorvpn xray-update скачать свежий xray-core
monitorvpn help        показать справку
monitorvpn delete      полностью удалить (plist + ~/.xrs-probe + symlink)
```

Обновлять агента при выходе новых версий — **без переустановки**: `monitorvpn update`
(токен и регистрация сохраняются). На Windows — `monitorvpn update` в PowerShell.

Если `/usr/local/bin/` без прав на запись — установщик подскажет точную `sudo ln -s …`
команду. До этого вызов можно делать через полный путь: `~/.xrs-probe/monitorvpn status`.

**Поставить пробника на Windows** (нужен Python 3.10+ — `winget install Python.Python.3.12`,
плюс `ADMIN_TOKEN` со страницы):

```powershell
# В PowerShell (не обязательно admin — пробник работает в твоей пользовательской сессии)
mkdir ~\xrs-probe; cd ~\xrs-probe
curl.exe -fsSL https://raw.githubusercontent.com/ASTORKA/xray-checker-statuspage/main/probes/install-windows.ps1 -o install-windows.ps1
.\install-windows.ps1
# спросит: URL статус-страницы, ADMIN_TOKEN, имя пробника
```

Что происходит:
- Регистрирует пробника на сервере (`mode=merge` — если уже был пробник с этим
  именем, переиспользуется его `probe_id`, история сохраняется).
- Кладёт `agent.py` и `monitorvpn.ps1` в `%USERPROFILE%\.xrs-probe\`.
- Создаёт **Scheduled Task `XrayCheckerProbe`** с триггером «при входе в систему»
  и авто-перезапуском при падении.
- Кладёт `monitorvpn.cmd` в `%USERPROFILE%\.local\bin\` и добавляет эту папку в
  User PATH (перезайди в PowerShell после установки).
- Логи: `%USERPROFILE%\.xrs-probe\agent.log`.

Команды `monitorvpn` те же, что на macOS. Снести: `monitorvpn delete` или
`.\install-windows.ps1 -Uninstall`.

**Поставить пробника на Linux** (готового установщика нет, но агент — один
кросс-платформенный `agent.py`, ставится вручную; нужен `python3` и `unzip`):

```bash
# 1) Зарегистрировать пробника на сервере (ADMIN_TOKEN со страницы) → probe_token:
TOKEN=$(curl -fsSL -X POST -H "X-Admin-Token: ВАШ_ADMIN_TOKEN" \
  -H "Content-Type: application/json" -d '{"name":"linux-box","mode":"merge"}' \
  https://status.example.com/api/admin/probes | python3 -c 'import sys,json;print(json.load(sys.stdin)["probe_token"])')

# 2) Скачать agent.py и xray-core (под нужную арку — amd64/arm64):
mkdir -p ~/.xrs-probe && cd ~/.xrs-probe
curl -fsSL https://raw.githubusercontent.com/ASTORKA/xray-checker-statuspage/main/probes/agent.py -o agent.py
curl -fL https://github.com/XTLS/Xray-core/releases/download/v25.12.8/Xray-linux-64.zip -o x.zip
unzip -o x.zip xray && chmod +x xray && rm x.zip

# 3) Запустить (демонизировать через systemd/tmux по вкусу):
STATUSPAGE_URL=https://status.example.com PROBE_TOKEN=$TOKEN python3 agent.py
```

Для автозапуска заверни шаг 3 в systemd-юнит с `Restart=always`.

**Обновление подписки на сервере.** Сервер тянет подписку в момент запроса от
пробника, кеширует на `PROBE_TARGETS_TTL_MIN` минут (по умолчанию 10). Когда
обновляешь подписку (новые сервера, или сменился `PROBE_SUBSCRIPTION_URL`), пробники
подхватят изменения автоматически в течение этого окна. Если нужно сейчас —
команда `monitorvpn refresh`: пробник пинает сервер с `?force=1`, кеш
инвалидируется, новый список приходит на следующем цикле агента.

**Что проверяет агент (через `xray-core`):**
- Установщик сам качает `xray-core` (пин на `v25.12.8`) и кладёт рядом
  (`~/.xrs-probe/xray`, на Windows `xray.exe`).
- Для каждого vless-сервера из подписки агент поднимает `xray-core` с тем же
  fingerprint/REALITY (`fp`, `pbk`, `sid`, `flow`), что и в подписке, и через локальный
  прокси:
  1. **дёргает cloudflare /cdn-cgi/trace** — туннель жив + внешний IP отличается от
     прямого (значит трафик реально пошёл через сервер);
  2. **бьёт пачкой параллельных коннектов к YouTube** (`PROBE_BURST_COUNT`, по умолчанию
     30). Это главный тест: одиночный коннект проходит даже на заблокированном
     fingerprint, а пачка хендшейков триггерит DPI-блок endpoint'а — ровно как у
     реального клиента под нагрузкой.
- Если прошло меньше `PROBE_BURST_OK_RATIO` (по умолчанию 0.5) коннектов — сервер
  помечается нерабочим с понятной причиной.
- При каждом цикле агент проверяет свою геолокацию (`ifconfig.co`). Если страна
  ≠ `EXPECT_COUNTRY` (по умолчанию `RU`) — на устройстве включён VPN: пробы не
  запускаются (через туннель они бессмысленны), а серверу шлётся отметка «был VPN» →
  на графике эти минуты серые.
- Если statuspage-сервер стал недоступен (например, после выключения VPN остаётся
  «отравленный» DNS) — агент сам перезапускается через пару циклов, без ручного
  `monitorvpn restart`.

## Частота опроса

`POLL_INTERVAL` (env `statuspage`) — период sync подписки в список серверов UI, по
умолчанию 60 сек. Интервал самих пробников задаётся на агенте через env `INTERVAL`
(по умолчанию 60 сек). Практический минимум — 60 сек: каждый цикл агент гоняет
`PROBE_BURST_COUNT` параллельных коннектов на сервер, давать меньше смысла мало.

## Как считается простой

Простой считается только по подряд идущим неудачным проверкам: 1 сбой → 0 (блип), 2 подряд → 1
интервал, 3 → 2 и т.д. Процент аптайма учитывает все проверки, поэтому блип виден в проценте и
в подсказке как «кратковременный сбой».

## Очистка истории опросов

Накопились неактуальные данные (например, после отладки) и графики/аптайм показывают
ерунду? В **админ-режиме** (иконка-замок в шапке → ввод `ADMIN_TOKEN`) под статистикой
появляется кнопка **«Очистить историю опросов»**. Она стирает все накопленные данные
пробников (графики, 30-дневный аптайм, VPN-периоды), но **сохраняет регистрации
устройств и список серверов** — агенты продолжат слать, данные начнут копиться заново.

Точечно: × напротив строки сервера удаляет конкретный сервер с его историей (если он
ещё в подписке — вернётся при следующем sync, поэтому полезно для чистки старых дублей).

Через API (для скриптов): `POST /api/admin/wipe-history` с заголовком
`X-Admin-Token: <токен>`.

## Приватность

`noindex, nofollow`, маскировка заголовка `Server`, рандомизация CSS-классов при каждом старте —
сложнее найти инстансы по шаблонным «тегам». Секретный URL подписки держит только сервер —
на устройства пробников он не попадает (агент получает уже разобранные таргеты).
